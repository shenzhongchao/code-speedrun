# Unit 5: 分组队列 — 并发控制与每组排队

## 用大白话说

这个单元就像一个餐厅的叫号系统：每桌（群组）同一时间只能有一个服务员（容器）在服务。如果餐厅服务员总数达到上限（默认 5 个），新来的桌就得排队等。服务员忙完一桌后，先看这桌有没有新需求，再去服务排队的下一桌。

## 背景知识

NanoClaw 的并发模型有两层约束：
1. **每组串行**: 同一个群组同一时间只有一个容器在运行。这避免了两个容器同时处理同一群组的消息导致重复回复。
2. **全局并发限制**: 所有群组加起来最多同时运行 N 个容器（默认 5）。这防止资源耗尽。

**为什么不用线程池？** 因为 NanoClaw 的"工作单元"是 Docker 容器，不是线程。容器的启动和销毁成本远高于线程，所以需要更精细的调度策略：容器启动后会保持空闲等待（idle），期间可以通过 IPC 接收新消息，避免频繁启停。

## 关键术语

- **GroupState**: 每个群组的运行时状态，包括是否活跃、是否空闲等待、待处理的消息和任务
- **activeCount**: 当前正在运行的容器总数，用于全局并发限制
- **waitingGroups**: 因并发限制而排队等待的群组列表
- **idleWaiting**: 容器完成当前工作后进入的"待命"状态，可以接收新消息而不需要重启
- **drain（排空）**: 容器完成后检查是否有积压的任务或消息需要处理
- **IPC 管道**: 向已运行的容器发送新消息的机制（写入文件，容器轮询读取）

## 这个单元做了什么

模拟 GroupQueue 的完整行为：
1. 消息入队和并发检查
2. 任务入队和优先级处理
3. 容器空闲等待和 IPC 管道
4. 完成后的排空逻辑（drain）
5. 全局并发限制和等待队列

## 关键代码走读

### 入队决策树 `enqueueMessageCheck()`
收到消息时的决策：
- 该群组已有活跃容器？→ 标记 `pendingMessages=true`，等容器完成后处理
- 全局并发已满？→ 加入 `waitingGroups` 队列
- 都没有？→ 立即启动容器

### IPC 管道 `sendMessage()`
如果群组已有活跃容器且不是任务容器，可以通过 IPC 文件直接发送新消息给它，而不需要等容器结束再启动新的。这大幅减少了容器启停次数。

### 排空逻辑 `drainGroup()`
容器完成后的优先级：
1. 先处理待执行的任务（任务不会被重新发现，消息会）
2. 再处理积压的消息
3. 都没有？检查全局等待队列，让其他群组的容器启动

## 练习

1. **修改并发限制**: 把 `MAX_CONCURRENT` 改为 1，观察所有群组如何排队
2. **添加优先级**: 修改 `waitingGroups` 为优先级队列，让主频道的消息优先处理
3. **用自己的话解释**: 为什么 `drainGroup()` 要先处理任务再处理消息？如果反过来会有什么问题？

## 调试指南

- **观察点**: 在 `enqueueMessageCheck()` 的三个分支处打断点，观察不同场景走哪个分支
- **常见问题**: 如果消息"丢失"，检查 `pendingMessages` 标志是否被正确设置和清除
- **状态检查**: 打印 `activeCount` 和 `waitingGroups.length`，确认并发状态
