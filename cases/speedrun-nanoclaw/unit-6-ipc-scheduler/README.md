# Unit 6: IPC 与调度器 — 文件系统通信与定时任务

## 用大白话说

这个单元有两个角色：
1. **IPC 监听器**就像一个信箱管理员：每秒检查一次每个群组的信箱（文件夹），有新信件（JSON 文件）就处理，处理完就删掉。
2. **任务调度器**就像一个闹钟管理员：每分钟检查一次所有闹钟（定时任务），到点了就叫醒对应的群组去干活。

## 背景知识

**为什么用文件系统做 IPC？** 容器内的智能体需要和主进程通信（发消息、创建定时任务、注册群组等），但容器是隔离的——没有网络连接到主进程。NanoClaw 的解决方案是：把 IPC 目录挂载到容器里，智能体写 JSON 文件，主进程轮询读取。这种方式简单、可靠、不需要额外的网络配置。

**安全模型**: 每个群组有自己的 IPC 目录（`data/ipc/{group-folder}/`）。主进程通过目录名确定消息来源，而不是信任文件内容中的身份声明。非主群组只能操作自己的资源，主群组拥有管理员权限。

## 关键术语

- **IPC (Inter-Process Communication)**: 进程间通信。这里指容器和主进程之间通过文件系统交换数据
- **轮询 (Polling)**: 定期检查是否有新数据，而非等待通知。IPC 每 1 秒轮询一次，调度器每 60 秒
- **cron 表达式**: Unix 风格的定时规则，如 `0 9 * * 1-5` 表示"工作日每天 9 点"
- **context_mode**: 任务的上下文模式。`isolated` 每次新建会话，`group` 复用群组的现有会话
- **sourceGroup**: IPC 消息的来源群组，由目录路径决定（不可伪造）

## 这个单元做了什么

1. 模拟 IPC 监听器处理三种消息类型：发送消息、创建任务、注册群组
2. 演示权限控制：非主群组不能跨组操作
3. 使用真实的 cron-parser 库解析 cron 表达式
4. 模拟调度器的轮询循环：检查到期任务 → 入队执行

## 关键代码走读

### IPC 消息类型
- `message`: 发送消息到指定聊天。非主群组只能发到自己的聊天
- `schedule_task`: 创建定时任务。非主群组只能为自己创建
- `pause_task` / `resume_task` / `cancel_task`: 管理任务。非主群组只能管理自己的
- `refresh_groups`: 刷新群组元数据。仅主群组可用
- `register_group`: 注册新群组。仅主群组可用

### 权限检查模式
每个操作都遵循同一模式：`if (isMain || resource.belongsTo === sourceGroup)`。主群组是超级管理员，其他群组只能操作自己的资源。身份由 IPC 目录路径决定，容器无法伪造。

### cron 调度
使用 `cron-parser` 库计算下次运行时间。支持三种调度类型：
- `cron`: 标准 cron 表达式（如 `0 9 * * 1-5`）
- `interval`: 固定间隔毫秒数（如 `3600000` = 1 小时）
- `once`: 一次性任务，指定 ISO 时间戳

## 练习

1. **添加新的 IPC 类型**: 实现 `update_task` 类型，允许修改任务的 prompt 或 schedule
2. **实现任务超时**: 如果任务执行超过 5 分钟，自动标记为 error
3. **用自己的话解释**: 为什么 IPC 的身份验证基于目录路径而不是文件内容？如果基于文件内容会有什么安全风险？

## 调试指南

- **观察点**: 在 `processTaskIpc()` 的 `switch` 语句处打断点，观察不同消息类型的处理流程
- **常见问题**: 如果 cron 任务不触发，检查 `next_run` 是否为过去的时间（必须 ≤ 当前时间才会被 `getDueTasks()` 返回）
- **状态检查**: 打印 `tasks` 数组，确认每个任务的 `status` 和 `next_run` 字段
