# Unit 1: 全局总览 — NanoClaw 端到端主流程

## 用大白话说

这个单元就像一个邮局的工作流程：信件（WhatsApp 消息）到达 → 分拣员查看收件人（触发词匹配）→ 把信送到对应的办公室（容器）→ 办公室处理后把回信送回去。

## 背景知识

NanoClaw 是一个"单进程"架构的 AI 助手。所有逻辑跑在一个 Node.js 进程里，但智能体（Claude）的实际执行被隔离在 Docker 容器中。这种设计的好处是：代码简单到你能完全理解，同时安全性由操作系统级别的容器隔离来保证。

核心设计模式是**轮询循环（Polling Loop）**：主进程每隔 2 秒检查一次数据库里有没有新消息，有的话就启动容器去处理。这比 WebSocket 推送简单得多，对于个人助手的消息量完全够用。

## 关键术语

- **JID (Jabber ID)**: WhatsApp 内部的聊天标识符。群聊以 `@g.us` 结尾，私聊以 `@s.whatsapp.net` 结尾
- **触发词 (Trigger)**: 默认是 `@Andy`，只有包含触发词的消息才会被处理（主频道除外）
- **主频道 (Main Channel)**: 你的私聊/自聊，拥有管理员权限，不需要触发词
- **分组 (Group)**: 每个 WhatsApp 群聊对应一个"分组"，有独立的文件系统和记忆
- **IPC (Inter-Process Communication)**: 容器和主进程之间通过文件系统交换数据的机制 
- **Session ID**: Claude Agent SDK 的会话标识，用于保持对话上下文

## 这个单元做了什么

用 stub 模拟了 NanoClaw 的完整主流程：
1. 初始化数据库和状态
2. 连接 WhatsApp（模拟）
3. 收到消息 → 存入数据库
4. 轮询循环检测新消息
5. 匹配触发词 → 格式化消息 → 启动容器
6. 容器返回结果 → 发送回复

所有外部依赖（WhatsApp、Docker、SQLite）都被替换为内存模拟，让你能在没有任何基础设施的情况下看到完整流程。

## 关键代码走读

### 入口 `main()` 函数
启动顺序：检查容器运行时 → 初始化数据库 → 加载状态 → 连接 WhatsApp → 启动调度器 → 启动 IPC 监听 → 启动消息轮询循环。每个子系统通过依赖注入（传入回调函数）解耦。

### 消息轮询循环 `startMessageLoop()`
每 2 秒执行一次：从数据库拉取新消息 → 按群组分组 → 检查触发词 → 尝试发送给已有容器（IPC 管道）→ 如果没有活跃容器则排队启动新容器。

### 消息处理 `processGroupMessages()`
这是核心处理函数：拉取该群组的未处理消息 → 格式化为 XML → 调用 `runAgent()` 启动容器 → 流式接收输出 → 发送回复。如果出错且还没发过回复，会回滚消息游标以便重试。

### 消息格式化
消息被格式化为 XML：`<messages><message sender="张三" time="...">内容</message></messages>`。这让 Claude 能清楚地区分不同发送者和时间。

## 练习

1. **修改触发词**: 把 `ASSISTANT_NAME` 改成你自己的名字，观察触发词匹配的变化
2. **添加第二个群组**: 在 `registeredGroups` 中注册第二个群组，模拟发送消息到不同群组
3. **用自己的话解释**: 为什么主频道（main channel）不需要触发词？这个设计决策背后的安全考量是什么？

## 调试指南

- **观察点**: 在 `processGroupMessages()` 的 `missedMessages` 变量处打断点，可以看到消息是如何被筛选的
- **常见问题**: 如果消息没被处理，检查 `lastAgentTimestamp` 是否已经超过了消息的时间戳（游标已前进）
- **状态检查**: 打印 `registeredGroups` 对象，确认 JID 格式正确（必须以 `@g.us` 或 `@s.whatsapp.net` 结尾）
